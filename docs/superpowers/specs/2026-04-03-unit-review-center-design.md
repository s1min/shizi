# 学习单元与单元复习设计

## 背景
当前 `shizi-frontend` 已经具备以下学习相关能力：
- 首页展示当前单元与后续单元，并可进入学习或测试
- 图鉴页展示已学汉字与今日复习入口
- 复习页支持“今日待复习字”模式
- 单元测试页支持测试并记录星级

但现状仍缺少一个明确的“家长管理入口”，让家长可以：
- 切换不同字库
- 查看该字库下全部单元的学习情况
- 从任意已学单元发起复习或测试

这导致产品存在两个问题：
1. 家长只能在首页围绕“当前单元”做决策，无法稳定回看历史单元
2. 现有“复习”能力是全局今日复习，不支持“只复习某个单元”

本次需求的目标不是重做学习主链路，而是在现有学习体系之上补齐一个家长侧的“学习单元”管理页，并将复习页扩展为支持“单元复习”模式。

## 用户与产品判断

### 目标用户
- 主用户：家长
- 间接服务对象：儿童

### 产品判断
本功能应被定义为“家长管理学习资产的入口”，而不是“孩子继续闯关的入口”。

因此：
- 主入口放在 `我的 > 家长中心`
- 页面语义定义为“学习单元”，而不是“复习中心”
- 页面职责是查看与调度，不承载儿童主学习任务

## 目标
- 新增家长可用的“学习单元”入口
- 支持按字库切换查看全部单元
- 支持四种单元状态的可视化区分
- 支持从单元卡片直接进入学习、单元复习、单元测试
- 已测试单元显示星标
- 复习页支持“单元复习”模式，默认复习该单元全部字
- 支持“常错字强化”模式，并按单元维度查看和进入强化任务
- 为后续更细分的强化策略预留扩展位

## 非目标
- 本次不重构学习页、测试页主交互
- 本次不新增独立的“单元复习记录”模型
- 本次不新增服务端“单元错字统计”聚合接口，先复用前端已有错题记录

## 方案对比

### 方案 A：入口放在 `我的 > 家长中心`，新增学习单元页，复习页扩展为多模式
这是推荐方案。

优点：
- 符合家长心智，不打断孩子首页主任务
- 单元浏览、字库切换、复习和测试调度语义清晰
- 复习能力可沉淀为统一页面的多模式架构，便于后续扩展错字强化

缺点：
- 需要同时修改前端页面、store 和后端接口

### 方案 B：入口放在图鉴页，做“汉字 / 单元”双 Tab
优点：
- 与“已学内容”相关性较强

缺点：
- 图鉴页当前偏孩子向收集视角，不适合承载家长管理页
- 容易把“内容回看”和“任务调度”混在一起

### 方案 C：新增 tabbar 作为一级入口
优点：
- 入口显眼

缺点：
- 破坏现有 `今日 / 图鉴 / 我的` 一级导航节奏
- 把家长管理能力抬升到孩子一级导航，不符合主次关系

### 结论
采用方案 A。

## 信息架构

### 入口位置
- 页面：`我的`
- 分组：`家长中心`
- 菜单名称：`学习单元`
- 菜单说明：`查看全部单元进度，选择复习或测试`

### 页面层级
`我的` -> `学习单元`

### 页面职责
“学习单元”页只负责：
- 查看指定字库下所有单元
- 看状态
- 做决策
- 跳转到学习、复习、测试

它不负责：
- 承担儿童主学习入口
- 替代首页当前单元卡
- 展示复杂学习报告

## 页面设计

### 页面名称
`学习单元`

### 顶部导航
- 左侧返回按钮
- 中间使用 `wd-tabs`，与返回按钮同一行
- Tab 1：`单元复习`
- Tab 2：`常错字强化`
- 不放右侧操作
- 前端优先使用 `wot-design-uni` 的 `wd-navbar`

### 字库切换区
- 位于导航下方第一块
- 展示当前字库名称
- 提示文案：`切换字库后，可查看该字库全部单元学习情况`
- 点击后使用 `wd-picker` 或 `wd-action-sheet` 进行切换

产品约束：
- 这里的切换是“查看维度切换”
- 同时同步 `learnStore.currentLibraryId`，以保持前后端状态一致

### 单元复习 Tab

#### 状态总览区
四个统计块：
- 未学习
- 学习中
- 待测试
- 已测试

每块展示：
- 状态名称
- 数量

作用：
- 帮助家长先快速建立全局进度感
- 再进入下方单元列表做具体决策

#### 单元列表区
- 按 `stage` 分组展示
- 每个 `stage` 默认展开，不做默认折叠
- 每个 `unit` 使用统一卡片展示

#### 单元卡片结构
每张卡片包含：
1. 标题区
2. 内容区
3. 操作区

##### 标题区
- 单元名称
- 状态标签
- 若已测试，显示星标

##### 内容区
- `lesson` 信息
- 汉字预览（最多显示前 6 个，多余显示 `+N`）
- 进度补充文案

##### 操作区
按钮规则如下：
- 未学习：`开始学习`
- 已学习未完成：`继续学习`
- 已学完未测试：`复习`、`开始测试`
- 已学完已测试：`复习`、`重新测试`

按钮优先级：
- 学习或测试按钮是主按钮
- `复习` 一律是次按钮
- 前端优先使用 `wd-button`

### 常错字强化 Tab

#### 产品语义
- 保持单元维度，不做全字库“大错题池”
- 家长先识别“哪个单元需要处理”，再进入强化
- 强化任务的目标是帮助家长处理“当前最该补的易错单元”

#### 顶部任务概览
四个统计块：
- 有错字单元
- 待强化错字
- 近 24 小时新错
- 当前主要薄弱点

#### 列表结构
- 默认展示 `待强化` 分组
- `已清空` 单元收纳到折叠区，默认不展开
- `待强化` 单元按错字数从高到低排序
- 若错字数相同，优先展示更靠前学习阶段和更接近测试的单元

#### 强化卡片信息
- 单元名称
- 待强化标签
- 错字数量
- 最近一次错题时间
- 主要错题来源类型
- 强化优先级徽标

#### 强化操作
- 有错字：`开始强化`
- 无错字：`暂无错字`
- 强化入口路由：
  - `/subpkg-learning/review/index?mode=unit&unitId={unitId}&scope=wrong`

## 单元状态设计

### 状态一：未学习
判定：
- `charIndex = 0`
- `learnCompleted = false`

展示：
- 状态标签：浅灰
- 文案：`尚未开始`
- 按钮：`开始学习`

### 状态二：学习中
判定：
- `charIndex > 0`
- `learnCompleted = false`

展示：
- 状态标签：浅蓝
- 文案：`已学 X/Y 字`
- 按钮：`继续学习`

### 状态三：待测试
判定：
- `learnCompleted = true`
- `testCompleted = false`

展示：
- 状态标签：浅橙
- 文案：`已学完，待测试`
- 按钮：`复习`、`开始测试`

### 状态四：已测试
判定：
- `testCompleted = true`

展示：
- 状态标签：浅绿
- 文案：`已完成测试`
- 星标：`1~3 星`
- 按钮：`复习`、`重新测试`

### 设计判断
- 不新增第五种状态
- 星标用于表达已测试单元的结果差异
- 四态模型保持稳定，降低理解成本

## 复习语义设计

### 核心判断
单元卡片上的 `复习` 和 `常错字强化` 都不能复用当前“今日复习”的语义。

`复习` 应被定义为：
- 单元复习
- 只针对当前单元
- 默认复习该单元全部字

`常错字强化` 应被定义为：
- 单元强化任务
- 只针对当前单元未重试错字
- 强化完成后回到列表时，需要给家长明确完成反馈

否则将产生认知错误：
- 家长点击某个单元的复习
- 却进入全局今日待复习池
- 页面结果与操作对象不一致

### 复习模式抽象
复习页统一抽象为多模式：
- `mode=today`
- `mode=unit`

本次实现：
- `mode=today`：沿用现有今日复习逻辑
- `mode=unit&unitId=xxx&scope=all`：取该单元全部字
- `mode=unit&unitId=xxx&scope=wrong`：取该单元未重试错字

后续扩展预留：
- `mode=wrong`
- 或 `mode=unit&scope=wrong`

推荐未来兼容写法：
- `mode=unit&unitId=xxx&scope=all`
- `mode=unit&unitId=xxx&scope=wrong`

本次直接开放两种 scope 的 UI：
- `scope=all`：单元复习
- `scope=wrong`：常错字强化

## 前端设计

### 页面新增
新增页面：
- `src/subpkg-learning/unit-list/index.vue`

页面职责：
- 渲染导航内 Tab
- 渲染字库切换区
- 渲染单元复习统计和列表
- 渲染常错字强化统计和任务列表
- 承接单元学习 / 单元复习 / 单元测试跳转

### 组件策略
前端实现要求：
- 能用 `wot-design-uni` 的组件就优先用
- 不重复造导航、按钮、标签、空态等基础 UI

建议组件：
- `wd-navbar`
- `wd-button`
- `wd-tag`
- `wd-loading`
- `wd-action-sheet` 或 `wd-picker`
- `wd-tabs`
- `wd-collapse`

### store 设计
基于现有 `src/store/learn.ts` 扩展，避免重建第二套学习数据源。

新增或扩展能力：
1. `setCurrentLibrary(libraryId: string)`
   - 设置当前字库
   - 加载对应字库详情
   - 加载对应汉字详情

2. `getUnitOverviewByLibrary(libraryId?: string)`
   - 返回指定字库下，按 stage 分组的单元视图模型
   - 每个 unit 包含：
     - `id`
     - `name`
     - `lesson`
     - `chars`
     - `charPreview`
     - `totalChars`
     - `charIndex`
     - `status`
     - `stars`
     - `learnCompleted`
     - `testCompleted`

3. `getUnitReviewChars(unitId: string)`
   - 返回该单元全部字
   - 供 `mode=unit` 的复习页使用

4. `getUnitWrongChars(unitId: string)`
   - 返回该单元未重试错字
   - 当前直接供错字强化 UI 使用

5. `resolveUnitIdByCharId(charId: string)`
   - 将历史空 `unitId` 的错题归一到实际单元
   - 用于兼容旧错题数据与复习场景新增错题

### 复习页改造
现有：
- `src/subpkg-learning/review/index.vue`

改造方向：
- 继续保留单页
- 增加 query 参数解析
- 根据模式加载复习字列表

支持模式：
1. `mode=today`
   - 使用现有 `todayReviewChars`

2. `mode=unit&unitId=xxx&scope=all`
   - 使用 `getUnitReviewChars(unitId)`

3. `mode=unit&unitId=xxx&scope=wrong`
   - 使用 `getUnitWrongChars(unitId)`

UI 文案建议：
- `today` 模式：标题 `今日复习`
- `unit + scope=all`：标题 `单元复习`
- `unit + scope=wrong`：标题 `错字强化任务`

### 页面跳转设计
- `开始学习` / `继续学习`
  - `/subpkg-learning/learn/index?unitId={unitId}`

- `复习`
  - `/subpkg-learning/review/index?mode=unit&unitId={unitId}&scope=all`

- `开始强化`
  - `/subpkg-learning/review/index?mode=unit&unitId={unitId}&scope=wrong`

- `开始测试` / `重新测试`
  - `/subpkg-learning/learn/unit-test?unitId={unitId}`

### 页面入口设计
修改：
- `src/pages/me/index.vue`

新增菜单项：
- 分组：`家长中心`
- 名称：`学习单元`
- 点击跳转至 `学习单元` 页面

## 后端设计

### 现有数据判断
当前 `shizi-backend` 已有学习数据结构：
- `charRecords`
- `unitProgressMap`
- `currentLibraryId`
- `stars`

这套结构已经足够支撑本次需求，不需要新增数据库表或文档模型。

### 新增接口
推荐新增：
- `GET /learning/unit-overview?libraryId=xxx`

### 返回内容
接口返回指定字库下的单元总览，建议包含：
- `libraryId`
- `libraryName`
- `stages`

每个 stage：
- `id`
- `name`
- `units`

每个 unit：
- `id`
- `name`
- `lesson`
- `chars`
- `totalChars`
- `charIndex`
- `learnCompleted`
- `testCompleted`
- `stars`
- `status`

### 服务端状态判定
后端新增统一状态判定逻辑，避免前后端语义漂移。

判定规则：
- `not_started`
- `learning`
- `ready_for_test`
- `tested`

推荐在 `learning.service.ts` 中封装统一方法。

### 为什么要增加聚合接口
虽然前端理论上可以基于字库详情和 `unitProgressMap` 自行拼装，但仍建议新增聚合接口，原因如下：
1. 字库切换与状态拼装放在后端更稳
2. 多端一致性更好
3. 前端页面职责更单纯
4. 后续若扩展错字强化、最近测试时间等字段，接口更容易演进

### 单元复习接口
本次不新增专门的“单元复习列表接口”。

原因：
- 单元复习当前定义为“该单元全部字”
- 字库详情已包含单元汉字列表
- 前端可直接基于单元 chars 构造复习内容

后续若出现以下需求，再考虑新增接口：
- 单元错字强化
- 按熟练度过滤
- 服务端排序推荐

## 技术实现顺序
推荐顺序：
1. 后端新增 `unit-overview` 接口
2. 前端扩展 `learnStore` 的字库切换与单元视图模型能力
3. 前端新增 `学习单元` 页面
4. `我的` 页面增加入口
5. 复习页支持 `mode=unit`
6. 联调验证状态、按钮与跳转

## 风险与边界

### 风险 1：`复习` 语义和 `今日复习` 冲突
约束：
- 单元卡片上的 `复习` 必须绑定单元模式
- 不允许直接跳到全局今日复习

### 风险 2：前端本地逻辑过重
约束：
- 通过后端 `unit-overview` 接口降低前端拼装复杂度

### 风险 3：错字归属不准确导致强化任务失真
约束：
- 复习答错时必须能回填所属单元
- 读取错题时需要兼容历史空 `unitId` 记录

### 风险 4：未来错字强化扩展受阻
约束：
- 当前复习模式设计必须预留 `scope` 或新 `mode`
- 不把单元复习逻辑写死在页面模板里

### 风险 5：字库切换和当前学习路径混淆
约束：
- 页面文案明确这是“查看与选择”
- 技术上允许同步 `currentLibraryId`，但页面语义仍以查看为主

## 验收标准
- `我的 > 家长中心` 中出现 `学习单元` 入口
- 点击后可进入新页面
- 页面支持切换字库
- 导航栏中间以 Tab 形式提供 `单元复习 / 常错字强化`
- 页面能展示全部单元，且按 stage 分组
- 四种状态显示正确
- 各状态按钮符合需求：
  - 未学习：`开始学习`
  - 学习中：`继续学习`
  - 待测试：`复习`、`开始测试`
  - 已测试：`复习`、`重新测试`
- 已测试单元显示星标
- 单元复习进入复习页时，只加载该单元全部字
- 常错字强化进入复习页时，只加载该单元未重试错字
- 常错字强化 Tab 按错字任务优先级排序
- 强化完成返回列表时，有明确轻反馈
- 现有“今日复习”逻辑不被破坏
- 首页“今日复习”可在新学后当天出现

## 结论
本次需求的正确落点不是单纯首页增强，而是补齐一个家长侧的“学习单元”管理页，并将复习能力升级为可区分“今日复习”“单元复习”“常错字强化”的统一系统。

最终方案为：
- 入口：`我的 > 家长中心 > 学习单元`
- 页面：导航同一行 Tab，支持 `单元复习 / 常错字强化`
- 动作：学习 / 单元复习 / 单元测试 / 常错字强化
- 复习：默认复习该单元全部字
- 强化：默认强化该单元未重试错字
- 扩展：后续可继续下沉到更细的错因分析与服务端聚合
