# 实机测试 Bug 记录规范

> 适用阶段：Phase C（提审准备）  
> 更新日期：2026-03-28

---

## 1. 目标

在实机测试阶段，保证每个问题都满足以下要求：

1. 可复现：有明确步骤和触发条件。
2. 可定位：有平台信息、接口信息、日志线索。
3. 可追踪：有唯一编号和状态流转。
4. 可回归：修复后能按同一用例复测并关闭。

---

## 2. 文件结构

1. `docs/bug_reports/_bug_report_template.md`  
单条缺陷模板，确认问题后复制一份并填写。
2. `docs/bug_reports/session_2026-03-28.md`  
当天实机测试台账，先快速记录，再拆分为单条缺陷文档。
3. `docs/device_regression_evidence.md`  
提审证据文档，引用 Bug ID 和截图/录屏。

---

## 3. 使用流程

1. 发现问题：先记到 `session_YYYY-MM-DD.md`，至少写平台、页面、现象、是否可稳定复现。
2. 问题确认：复制 `_bug_report_template.md` 生成单条缺陷文档，补齐复现步骤和证据。
3. 开始修复：状态改为 `FIXING`，填写计划修复版本和负责人。
4. 修复完成：状态改为 `FIXED`，补充代码提交信息和验证说明。
5. 回归通过：状态改为 `VERIFIED` 或 `CLOSED`，在会话台账中同步结果。

---

## 4. 状态定义

1. `NEW`：新发现，尚未确认根因。
2. `CONFIRMED`：已确认可复现且可定位范围。
3. `FIXING`：已进入修复。
4. `FIXED`：代码已改，等待回归。
5. `VERIFIED`：回归通过，待归档。
6. `CLOSED`：关闭完成。
7. `BLOCKED`：被外部条件阻塞（如第三方服务、网络策略）。
8. `REJECTED`：非缺陷或不处理（需写明原因）。

---

## 5. 严重级别

1. `S0`：阻塞提审或核心链路不可用（登录失败、白屏、数据错乱）。
2. `S1`：核心功能可用但明显异常（高频报错、结果错误、严重体验问题）。
3. `S2`：非核心异常或低频问题（文案、样式、轻微交互问题）。

---

## 6. 命名规范

单条缺陷文档建议命名：

`BUG-YYYYMMDD-序号-简短英文标识.md`

示例：

`BUG-20260328-001-wx-login-401.md`

---

## 7. 给我修复时的最小信息集

每次把 Bug 交给我修时，建议直接提供：

1. Bug ID（例如 `BUG-20260328-001`）
2. 复现步骤（最少 3 步）
3. 预期结果和实际结果
4. 环境信息（iOS/Android、微信版本、接口 URL）
5. 证据路径（截图/录屏/日志）

这样我可以直接进入定位和修复，不需要反复追问上下文。
