# 产品原型设计方案 v3：个人开发者落地版

> 基于 v2 方案，从个人开发者实际可执行角度进行精简优化。
> 原则：**能砍则砍，能推迟则推迟，聚焦核心学习闭环。**

---

## 一、v2 → v3 优化总览

### 1.1 核心调整原则

作为个人开发者，必须遵循以下原则：
- **一个人 5 周内能做完** — 任何超出此范围的功能一律推迟
- **技术栈与现有代码对齐** — 不做无谓的技术迁移
- **功能做减法** — MVP 只保留让用户完成"学一个单元"的最短路径
- **设计做减法** — 不追求完美交互，先跑通再优化

### 1.2 关键变更一览

| 维度 | v2 方案 | v3 调整 | 理由 |
|------|---------|---------|------|
| 数据库 | PostgreSQL + JSONB | **MongoDB**（保持现状） | 项目已用 MongoDB，迁移浪费时间 |
| 底部导航 | 4 Tab | **3 Tab**（今日/图鉴/我的） | 星星商店 MVP 不做，少一个 Tab 少一个页面 |
| 星星商店 | 完整兑换系统 | **砍掉** → P2 | 一个人做不了虚拟货币+商品+兑换+动画 |
| 魔法造句 | AI 造句功能 | **砍掉** → P2 | 需要接 LLM API，成本和复杂度都高 |
| 自适应学习 | 按年龄/表现跳步 | **简化为固定流程** | 自适应逻辑复杂，先用固定 4 步验证产品 |
| 亲子模式 | 独立功能模块 | **砍掉** → P2 | 锦上添花，不影响核心体验 |
| 语音跟读 | ASR 语音评测 | **降级为点击选择** | 语音识别在小程序端坑多，MVP 先跳过 |
| 打卡海报 | Canvas 绘制+分享 | **简化为结算页** | Canvas 海报生成耗时，先做简单结算 |
| 新用户引导 | 完整 4 步引导流 | **简化为 2 步** | 选年龄 + 直接开始学习 |
| 部署 | Docker Compose + Nginx | **轻量云 + Docker Compose** | 不绑定云平台，用户增长后可平滑迁移 |

## 二、产品概述（精简版）

- **产品名称**：趣字宝 / QuZiBao
- **平台**：微信小程序（MVP 唯一平台）
- **目标用户**：3-6 岁学龄前儿童及其家长
- **MVP 核心体验**：用户能完整学完一个单元（6 个汉字），包含字源动画、认读、描红、测验，并保存进度
- **核心差异化**：字源动画教学（这一个点做好就够了）

## 四、详细页面设计

### 4.1 新用户引导（简化版）

**v2 是 4 步，v3 简化为 2 步**：

1. **微信授权登录**（一键）
2. **选择字库** → 直接进入首页

**砍掉的部分**：
- 不做"体验第一个字"引导（用户进首页后自然会点"开始学习"）
- 不做年龄段推荐（让用户自己选字库，选错了可以随时切换）
- 不做宝宝档案创建引导（在"我的"页面随时可以设置）

**理由**：引导流程做得再精致，用户也是点两下就过了。把精力放在核心学习体验上。

---

### 4.2 首页：今日任务

**保留 v2 的任务卡片式设计**，这个方向是对的。简化实现：

**布局**：

- **顶部**：宝宝昵称 + 连续打卡天数 + 已学字数
- **卡片 1：复习任务**（有待复习字时显示）
  - "今天有 X 个字需要复习"
  - 按钮："开始复习"
- **卡片 2：新字学习**
  - 当前字库 + 当前单元 + 进度条
  - 按钮："开始学习"
- **底部**：字库切换入口

**MVP 砍掉的**：
- 不做"魔法造句"卡片
- 不做复杂的动画效果
- 连续打卡用简单数字显示，不做火焰动画

---

### 4.3 核心学习流程（固定 4 步）

**v2 的自适应跳步机制砍掉**，MVP 统一走固定 4 步：

```
字源动画 → 认读练习 → 描红练习 → 即时测验
```

每个字完成 4 步后进入下一个字，单元内所有字完成后进入单元闯关。

#### 步骤 1：字源动画

保留 v2 设计，这是产品核心差异化，必须做好。

- 按汉字类型分层（象形/会意/抽象）
- 动画时长 8-12 秒
- 动画结束自动进入下一步
- "再看一遍"按钮

**MVP 实现方式**：
- 象形字：图片 → 汉字的渐变动画（CSS/Canvas 实现）
- 会意字：两个部件组合动画
- 抽象字：简单的文字说明 + 读音播放（不做复杂动画）

#### 步骤 2：认读练习（降级版）

**v2 设计的语音跟读 → 降级为点击选择**

- 屏幕中央：大字展示 + 拼音
- 系统自动播放读音（TTS）
- **交互**：播放 3 个读音选项，用户点击选择正确的读音
- 答对：进入下一步；答错：重新播放正确读音，再选一次

**为什么砍语音跟读**：
- 微信小程序的录音 + ASR 识别链路坑很多（权限、兼容性、准确率）
- 个人开发者调试这个至少要 1-2 周
- 点击选择同样能达到"听音辨字"的教学目的

#### 步骤 3：描红练习

保留 v2 设计，使用 HanziWriter 库。

- 笔顺引导 + 描红交互
- 容错度适当放宽
- 完成后自动进入下一步

**MVP 注意**：
- 如果 HanziWriter 在小程序端适配困难，降级为"笔顺动画展示"（只看不写），不要在这里卡太久

#### 步骤 4：即时测验

保留 v2 的多题型设计，但简化题型选择逻辑：

- MVP 阶段只实现 **2 种题型**：
  - 题型 A：看字选图（给汉字，选对应图片）
  - 题型 B：听音选字（播放读音，选对应汉字）
- 每个字出 1 道题，随机选题型
- 答对得 1 颗星星，答错高亮正确答案 + 重播读音

**砍掉的题型**：
- 题型 C（看图选字）：与题型 A 高度重复，MVP 不需要
- 题型 D（语境选字）：需要大量例句数据，推迟到 P2

#### 步骤 5：单元闯关

- 单元内所有字学完后触发
- 题目数 = 单元字数，每字一题
- 80% 正确率通过，获得 1-3 星评级
- 错题加入复习队列

#### 步骤 6：结算页

**v2 的打卡海报 → 简化为结算展示页**

- 显示：本次学习字数、正确率、获得星星数
- "完成"按钮返回首页
- **不做**：Canvas 海报生成、分享功能（推迟到 P1）

---

### 4.4 复习系统

保留 v2 的闪卡复习设计，这个交互简单且有效。

#### 闪卡模式
- 全屏卡片，左右滑动
- 正面：大字（无拼音）
- 点击翻转：拼音 + 组词
- 底部："认识" / "不认识"
- 复习等级按艾宾浩斯间隔调整

#### 复习间隔（与 v2 一致）

| 等级 | 间隔 |
|:---:|:---:|
| 0 | 当天再次出现 |
| 1 | 1 天 |
| 2 | 2 天 |
| 3 | 4 天 |
| 4 | 7 天 |
| 5 | 15 天 |
| 6 | 30 天 |
| 7 | 已掌握 |

---

### 4.5 汉字图鉴

简化版：

- 顶部：已学 X / 总 X 字（进度条）
- 网格展示所有汉字
  - 已掌握：彩色卡片
  - 学习中：白色卡片 + 进度标记
  - 未学习：灰色 + "?"
- 点击已学汉字：弹窗显示拼音、组词、可重播字源动画

**砍掉的**：魔法造句入口（P2）

---

### 4.6 我的

- **宝宝信息**：昵称、头像（微信头像）、已学字数
- **学习数据**：今日新学 X 字 / 复习 X 字 / 学习 X 分钟
- **设置**：
  - 每日新字数（3/5/8 可选）
  - 学习提醒时间
  - 当前字库切换
- **退出登录**

**砍掉的**：
- 等级称号系统（P2）
- 头像框兑换（P2）
- 亲子模式开关（P2）
- 家长安全锁（P2，MVP 阶段设置页不需要锁）

## 六、交互设计原则（精简版）

保留最关键的 4 条：

1. **大触控区**：按钮最小 88rpx × 88rpx，间距 24rpx+
2. **即时反馈**：每次操作都有视觉/声音反馈
3. **无文字依赖**：按钮用图标 + 文字
4. **退出保护**：学习中按返回弹确认框

**砍掉的**：
- "单手可操作"约束（增加布局复杂度，MVP 不必要）
- "音频优先"（需要录制大量语音，MVP 用 TTS 即可）

## 七、技术方案（与实际项目对齐）

### 7.1 技术栈（以现有代码为准）

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 前端框架 | uni-app + Vue 3 | 已搭建 |
| 状态管理 | Pinia（persist） | 已搭建 |
| 后端框架 | NestJS 11 | 已有脚手架 |
| 数据库 | **MongoDB** | **已选定，不迁移到 PostgreSQL** |
| ORM | Mongoose | 与 NestJS 配合成熟 |
| 鉴权 | JWT + 微信登录 | 前端 token store 已实现 |
| 部署 | Docker Compose + 轻量云 | 不绑定平台，可平滑扩展 |

**与 v2 的关键差异**：
- v2 提议迁移到 PostgreSQL + Prisma，但项目已经选定 MongoDB 且 project_plan 中明确使用 MongoDB Atlas。迁移数据库对 MVP 没有任何收益，纯粹浪费时间。
- MongoDB 的文档模型天然适合存储汉字元数据（teaching、metadata 等灵活结构），不需要 JSONB 的 workaround。

```javascript
    example_sentences: [
```

#### Collection 2: `libraries` — 字库编排


#### Collection 3: `users` — 用户

```javascript
{
  _id: ObjectId,
  openid: "wx_xxx",        // 微信 openid，唯一索引
  nickname: "小明",
  avatar: "url",
  created_at: Date,
```

**简化说明**：v2 有独立的 `children` 表支持多宝宝，MVP 阶段直接内嵌到 user 文档中。一个微信号 = 一个宝宝，够用了。

#### Collection 4: `learning_records` — 学习记录

```javascript
{
  _id: ObjectId,
  user_id: ObjectId,
  // 每个字的学习进度
  char_records: {
    "火": {
      learned_at: Date,
      review_level: 3,        // 艾宾浩斯等级 0-7
      next_review_at: Date,
      quiz_correct_count: 5,
      quiz_total_count: 6
    }
  },
  // 单元完成状态
  unit_progress: {
    "lib_1a_upper:unit_1": {
      completed: true,
      stars: 3,
      completed_at: Date
    }
  },
  // 学习天数记录（用于打卡统计）
  learn_days: ["2026-03-01", "2026-03-02"],
  updated_at: Date
}
```

**简化说明**：
- v2 有独立的 `user_progress`、`learning_records`、`achievements`、`settings` 四张表
- v3 合并为一个 `learning_records` 集合，用内嵌文档存储所有进度数据
- MongoDB 的文档模型非常适合这种"一个用户的所有学习数据"的聚合查询模式
- MVP 阶段数据量极小（每个用户最多几百个字的记录），单文档完全够用

### 7.3 后端 API（最小集）

只实现 MVP 必需的接口：

| 方法 | 路径 | 说明 | 优先级 |
|------|------|------|--------|
| POST | `/auth/wxLogin` | 微信登录 | P0 |
| GET | `/user/info` | 用户信息 | P0 |
| PUT | `/user/child` | 更新宝宝信息 | P1 |
| GET | `/libraries` | 字库列表 | P0 |
| GET | `/libraries/:id` | 字库详情（含单元和字列表） | P0 |
| GET | `/characters/batch` | 批量获取汉字数据（query: ids=火,水,山） | P0 |
| GET | `/learning/progress` | 获取学习进度 | P0 |
| PUT | `/learning/progress` | 全量更新学习进度 | P0 |
| GET | `/learning/review-list` | 今日待复习字列表 | P1 |

**总共 9 个接口**，比 v2 的 16 个接口砍掉近一半。

**砍掉的接口**：
- `/progress/:childId/today` — 前端本地计算即可
- `/shop/*` — 星星商店砍掉
- `/achievements/*` — 成就系统推迟
- `/records/:childId/daily` — 学习日报前端本地统计
- `/children` CRUD — 不支持多宝宝

### 7.4 部署方案

**轻量云服务器 + Docker Compose**，代码不绑定任何云平台：

```yaml
# docker-compose.yml
version: '3.8'
services:
  app:
    build: ./server
    ports:
      - "3000:3000"
    environment:
      - MONGODB_URI=mongodb://mongo:27017/hanzi
      - JWT_SECRET=${JWT_SECRET}
      - WX_APPID=${WX_APPID}
      - WX_SECRET=${WX_SECRET}
    depends_on:
      - mongo

  mongo:
    image: mongo:7
    volumes:
      - mongo_data:/data/db
    ports:
      - "27017:27017"

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/conf.d:/etc/nginx/conf.d
      - ./certbot/conf:/etc/letsencrypt
    depends_on:
      - app

volumes:
  mongo_data:
```

**方案选择理由**：
- **不用微信云托管**：厂商锁定严重，API 调用方式和数据库访问都是微信专有的，后期迁移痛苦；用户量增长后定价不透明、扩展性受限（WebSocket、定时任务等）
- **用轻量云 + Docker**：代码标准化，可在任何云平台运行；后期用户增长可平滑迁移到更大实例或 K8s
- **运维成本可控**：腾讯云/阿里云 2C2G 轻量云约 50-100 元/月，Docker Compose 一键启停，日常运维就是 `docker compose up -d`

**扩展路径**：
- 初期：单台 2C2G 轻量云，支撑 ~1000 日活
- 中期：升配到 4C4G 或加 MongoDB Atlas 专用集群
- 后期：迁移到 K8s，前后端分离部署，数据库独立托管

## 八、字库设计（与 v2 一致）

MVP 只上线 2 个字库：

| 字库 ID | 名称 | 字数 | 说明 |
|---------|------|------|------|
| `lib_preschool` | 学前启蒙 | ~200 | 按难度编排，3-5 岁 |
| `lib_1a_upper` | 一年级上册 | ~300 | 同步人教版教材，6 岁+ |

其余字库（一年级下册、二年级上下册）在"字库列表"中灰色展示"即将上线"。

## 十一、v2 → v3 砍掉的功能及恢复条件

| 功能 | 恢复条件 | 预估工时 |
|------|----------|----------|
| 星星商店 | MVP 上线后用户留存 > 30% | 2 周 |
| 魔法造句 | 累计用户 > 500，有预算接 LLM API | 1 周 |
| 语音跟读 | 微信 ASR 插件稳定性验证通过 | 1 周 |
| 自适应学习 | 收集到足够的用户学习行为数据 | 1 周 |
| 亲子模式 | 用户调研显示家长有强需求 | 1 周 |
| 打卡海报 | 需要社交传播拉新时 | 3 天 |
| 等级称号 | 星星商店上线后 | 3 天 |

**原则**：每个被砍的功能都有明确的恢复条件，不是"不做"而是"现在不做"。

## 十二、总结

v3 相比 v2 的核心变化：

| 维度 | v2（理想版） | v3（落地版） |
|------|-------------|-------------|
| 功能数量 | ~20 个功能点 | **~10 个功能点** |
| 后端接口 | 16 个 | **9 个** |
| 数据库表 | 7 张 | **4 个 Collection** |
| 底部 Tab | 4 个 | **3 个** |
| 题型数量 | 4 种 | **2 种** |
| 学习步骤 | 自适应 2-6 步 | **固定 4 步** |
| 部署方式 | Docker 自建 | **轻量云 + Docker Compose** |
| 开发周期 | 未评估 | **5 周** |

**一句话总结**：v3 = v2 的核心学习流程 + 最小可用的用户体系，砍掉一切不影响"学会一个字"的功能。先上线，再迭代。
